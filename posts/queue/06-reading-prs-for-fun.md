# Reading pull requests for fun

Twice a week I open a random pull request on a project I use. I do not comment. I just read.

Why? Because finished code teaches you nothing about how it was made. Pull requests show the conversation: someone proposes, someone pushes back, the patch changes shape, and finally it lands.

I learn three things from each one:

1. **How experienced people write commit messages.** Most are short. None are clever.
2. **How they answer "why" questions.** Often by linking an issue, not by writing a wall of text.
3. **What gets rejected and why.** This is the most useful part. The patches that *almost* shipped show me the line between "okay" and "yes".

Pick a project. Open the closed-PR list. Sort by recently merged. Read three. You will be a slightly better engineer by Friday.

Free school, every day.
